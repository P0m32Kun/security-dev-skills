# 用户视角功能验证

> **核心理念**：编译通过 ≠ 功能实现

## 前置条件
- 代码已完成（Implement 阶段通过）
- 单元测试和集成测试通过
- 文档已同步（Doc-Sync 阶段通过）

## 流程

### 1. 列出用户验收条件

从功能规格文档中提取验收条件，转换为可验证的检查项：

```markdown
## 用户验收条件

### 功能性验收
- [ ] 用户可以正常登录
- [ ] 登录后显示用户信息
- [ ] 登录态保持 24 小时
- [ ] 退出登录后跳转到登录页

### 非功能性验收
- [ ] 登录响应时间 < 2 秒
- [ ] 支持移动端访问
- [ ] 错误信息清晰易懂
```

### 2. 执行验证

#### 自动化验证（E2E 测试）

```bash
# 运行相关 E2E 测试
npx playwright test tests/e2e/login.spec.ts

# 查看测试结果
npx playwright show-report
```

#### 手动验证

对于无法自动化的场景，进行手动验证：

| 验证项 | 操作步骤 | 预期结果 | 实际结果 |
|--------|---------|---------|---------|
| 登录成功 | 输入正确账号密码 | 跳转到首页 | — |
| 密码错误 | 输入错误密码 | 显示错误提示 | — |
| 移动端访问 | 手机浏览器访问 | 页面正常显示 | — |

### 3. 输出验证报告

```markdown
## 功能验证报告

### 验证概要
- 功能：用户认证模块
- 验证时间：2026-06-02
- 验证人：xxx

### 验证结果

#### 自动化验证
- [x] 登录成功 — E2E 测试通过
- [x] 密码错误 — E2E 测试通过
- [ ] 登录态保持 — E2E 测试失败（见下方问题）

#### 手动验证
- [x] 移动端访问 — 页面正常显示
- [x] 错误信息 — 清晰易懂

### 问题记录

#### 问题 1：登录态保持失败
- **现象**：关闭浏览器后重新打开，需要重新登录
- **预期**：登录态应保持 24 小时
- **影响**：用户体验
- **建议**：检查 cookie 过期时间设置

### 验证结论
- 通过：8/10
- 失败：2/10
- **建议**：修复问题后重新验证
```

## 完成标准
- [ ] 所有用户验收条件已列出
- [ ] 自动化验证已执行
- [ ] 手动验证已完成（如需要）
- [ ] 验证报告已输出
- [ ] 所有验收条件满足（或问题已记录并有解决方案）
