# 标准化 Bug 修复流程

> 强制执行：原因分析→方案制定→计划编写→分支创建→实施→测试→PR→Review→合并 流程。

## 核心原则

1. **先诊断后修复** — 不分析根因不改代码
2. **先测试后提交** — 没有回归测试不合并
3. **最小变更** — 只修复问题，不做无关重构
4. **可追溯** — 每个步骤都有记录

## 流程总览

```
Diagnose → Plan → Implement → Test → PR → Review → Merge
    │         │         │         │      │       │       │
 根因分析   制定方案   编码修复   测试验证  提交PR  代码审查  合并代码
```

## 阶段详情

### 阶段 1：Diagnose（诊断）

**目标**：完成根因分析，明确问题本质

**输入**：Bug 描述（现象、复现步骤、期望行为）

**输出**：根因分析报告

<HARD-GATE>
**阻断条件**：未完成根因分析不得进入 Plan 阶段

**检查项**：
- [ ] Bug 现象已明确描述
- [ ] 复现步骤已确认
- [ ] 问题代码位置已定位
- [ ] 根本原因已分析清楚
- [ ] 影响范围已评估
</HARD-GATE>

**流程**：
1. **理解 Bug 现象**
   - 记录 Bug 的具体表现
   - 确认复现步骤
   - 确认期望行为 vs 实际行为

2. **定位问题代码**
   - 使用日志、堆栈跟踪定位
   - 使用代码搜索工具（Semble、CodeGraph）辅助定位
   - 缩小问题范围到具体函数/模块

3. **分析根本原因**
   - 使用 5 Why 分析法追溯根因
   - 区分直接原因和根本原因
   - 分析是代码逻辑错误、边界条件、并发问题还是设计缺陷

4. **评估影响范围**
   - 哪些功能受影响
   - 哪些用户受影响
   - 是否有数据损坏风险
   - 是否需要紧急修复

**输出格式**：
```markdown
## 根因分析报告

### Bug 现象
- **表现**：[具体现象描述]
- **复现步骤**：[1. 2. 3.]
- **期望行为**：[应该是什么样]
- **实际行为**：[实际是什么样]

### 问题定位
- **文件**：[path/to/file.ext]
- **函数/方法**：[function_name]
- **代码位置**：[行号范围]

### 根本原因
- **直接原因**：[导致错误的直接代码]
- **根本原因**：[为什么会写出这样的代码/设计]
- **问题类型**：[逻辑错误/边界条件/并发问题/设计缺陷]

### 影响范围
- **受影响功能**：[功能列表]
- **受影响用户**：[用户范围]
- **数据风险**：[是否有数据损坏风险]
- **紧急程度**：[critical/high/medium/low]
```

**Anti-Patterns**：
- ✗ 不分析根因，直接开始改代码
- ✗ 只看表面现象，不深挖根本原因
- ✗ 只定位到文件，不精确定位到函数/行号
- ✗ 不评估影响范围，盲目修复

---

### 阶段 2：Plan（计划）

**目标**：制定修复方案和实施计划

**输入**：根因分析报告

**输出**：修复计划

<HARD-GATE>
**阻断条件**：未制定修复计划不得进入 Implement 阶段

**检查项**：
- [ ] 修复方案已确定
- [ ] 影响范围已评估
- [ ] 测试策略已制定
- [ ] 回滚方案已准备
</HARD-GATE>

**流程**：
1. **制定修复方案**
   - 基于根因分析，确定修复策略
   - 评估多种方案的优劣
   - 选择最小风险的方案

2. **评估影响范围**
   - 修复会影响哪些功能
   - 是否需要修改接口
   - 是否需要数据迁移

3. **制定测试策略**
   - 如何验证修复成功
   - 如何确保不引入新问题
   - 需要哪些回归测试

4. **准备回滚方案**
   - 如果修复失败如何回滚
   - 回滚步骤是什么

**输出格式**：
```markdown
## 修复计划

### 修复方案
- **策略**：[修复策略描述]
- **原因**：[为什么选择这个方案]

### 变更清单
- **文件 1**：[path/to/file] — [变更描述]
- **文件 2**：[path/to/file] — [变更描述]

### 测试策略
- **单元测试**：[需要添加/修改的测试]
- **集成测试**：[需要添加/修改的测试]
- **回归测试**：[需要运行的现有测试]

### 回滚方案
- **回滚步骤**：[1. 2. 3.]
- **回滚验证**：[如何确认回滚成功]
```

**Anti-Patterns**：
- ✗ 不制定计划，直接开始改代码
- ✗ 不评估影响范围
- ✗ 不准备回滚方案
- ✗ 选择最复杂的修复方案

---

### 阶段 3：Implement（实施）

**目标**：按照修复计划实施修复

**输入**：修复计划

**输出**：修复后的代码

<HARD-GATE>
**阻断条件**：修复代码未完成不得进入 Test 阶段

**检查项**：
- [ ] 所有计划的变更已完成
- [ ] 代码符合项目规范
- [ ] 没有引入新的编译错误
</HARD-GATE>

**流程**：
1. **创建修复分支**
   ```bash
   git checkout -b fix/[bug-id]-[brief-description]
   ```

2. **实施修复**
   - 按照修复计划逐项实施
   - 每完成一项，标记为已完成
   - 遵循项目编码规范

3. **添加测试**
   - 添加复现 Bug 的测试
   - 添加验证修复的测试
   - 确保测试在修复前失败，修复后通过

4. **自测**
   - 运行相关测试
   - 手动验证修复效果
   - 确认没有引入新问题

**Anti-Patterns**：
- ✗ 跳过测试，直接改代码
- ✗ 修改不相关的代码
- ✗ 不遵循项目编码规范
- ✗ 不运行测试就提交

---

### 阶段 4：Test（测试）

**目标**：验证修复的正确性和完整性

**输入**：修复后的代码

**输出**：测试报告

<HARD-GATE>
**阻断条件**：测试不通过不得进入 PR 阶段

**检查项**：
- [ ] 复现测试通过（验证 Bug 已修复）
- [ ] 单元测试通过
- [ ] 集成测试通过
- [ ] 回归测试通过
</HARD-GATE>

**流程**：
1. **运行复现测试**
   - 确认 Bug 已修复
   - 确认修复是有效的

2. **运行单元测试**
   - 运行修改文件的单元测试
   - 运行相关模块的单元测试

3. **运行集成测试**
   - 运行相关功能的集成测试
   - 确认功能正常工作

4. **运行回归测试**
   - 运行全量测试（如果时间允许）
   - 确认没有引入新问题

5. **手动验证**
   - 按照复现步骤验证
   - 测试边界情况
   - 确认用户体验正常

**输出格式**：
```markdown
## 测试报告

### 测试结果
- 复现测试：✅ 通过
- 单元测试：✅ 通过 (X/X)
- 集成测试：✅ 通过 (X/X)
- 回归测试：✅ 通过 (X/X)

### 手动验证
- [ ] 按复现步骤验证通过
- [ ] 边界情况测试通过
- [ ] 用户体验正常
```

**Anti-Patterns**：
- ✗ 只运行部分测试
- ✗ 跳过手动验证
- ✗ 不测试边界情况
- ✗ 测试失败但强行继续

---

### 阶段 5：PR（提交 Pull Request）

**目标**：提交 PR，准备代码审查

**输入**：测试通过的代码

**输出**：Pull Request

<HARD-GATE>
**阻断条件**：PR 未创建不得进入 Review 阶段

**检查项**：
- [ ] PR 描述完整
- [ ] 关联的 Issue 已链接
- [ ] 变更清单已列出
- [ ] 测试结果已附上
</HARD-GATE>

**流程**：
1. **准备提交信息**
   ```bash
   git add .
   git commit -m "fix: [简短描述]

   - 修复 [问题描述]
   - 根本原因：[根因]
   - 解决方案：[方案]

   Fixes #[issue-id]"
   ```

2. **推送分支**
   ```bash
   git push origin fix/[bug-id]-[brief-description]
   ```

3. **创建 PR**
   - 填写 PR 模板
   - 关联 Issue
   - 添加标签

4. **PR 描述模板**
   ```markdown
   ## 问题描述
   [Bug 现象描述]

   ## 根本原因
   [根因分析]

   ## 解决方案
   [修复方案描述]

   ## 变更清单
   - [ ] 文件 1：变更描述
   - [ ] 文件 2：变更描述

   ## 测试
   - [ ] 复现测试通过
   - [ ] 单元测试通过
   - [ ] 集成测试通过
   - [ ] 手动验证通过

   ## 关联 Issue
   Fixes #[issue-id]
   ```

**Anti-Patterns**：
- ✗ PR 描述不完整
- ✗ 不关联 Issue
- ✗ 不附上测试结果
- ✗ 一个 PR 包含多个不相关的修复

---

### 阶段 6：Review（代码审查）

**目标**：进行代码审查，确保代码质量

**输入**：Pull Request

**输出**：审查意见

<HARD-GATE>
**阻断条件**：审查未通过不得进入 Merge 阶段

**检查项**：
- [ ] 代码逻辑正确
- [ ] 测试覆盖充分
- [ ] 没有安全问题
- [ ] 符合项目规范
</HARD-GATE>

**流程**：
1. **代码逻辑审查**
   - 修复是否正确解决了根因
   - 是否有遗漏的边界情况
   - 是否有潜在的副作用

2. **测试审查**
   - 测试是否充分
   - 是否覆盖了关键场景
   - 测试是否清晰易懂

3. **安全审查**
   - 是否引入安全漏洞
   - 是否有敏感信息泄露
   - 是否有权限问题

4. **规范审查**
   - 代码风格是否一致
   - 命名是否清晰
   - 注释是否充分

**输出格式**：
```markdown
## 审查意见

### 总体评价
[总体评价]

### 问题
- **严重**：[问题描述] — 必须修复
- **建议**：[问题描述] — 建议修复
- **可选**：[问题描述] — 可选修复

### 优点
- [做得好的地方]
```

**Anti-Patterns**：
- ✗ 审查不仔细，走形式
- ✗ 只关注代码风格，不关注逻辑
- ✗ 不给出具体的改进建议
- ✗ 审查周期太长

---

### 阶段 7：Merge（合并）

**目标**：合并修复代码，关闭 Issue

**输入**：审查通过的 PR

**输出**：合并后的代码

<HARD-GATE>
**阻断条件**：审查未通过不得合并

**检查项**：
- [ ] 审查意见已处理
- [ ] CI/CD 通过
- [ ] 没有合并冲突
</HARD-GATE>

**流程**：
1. **处理审查意见**
   - 修复审查提出的问题
   - 回复审查意见
   - 更新代码

2. **确认 CI/CD**
   - 所有检查通过
   - 没有新的警告

3. **合并代码**
   ```bash
   # 方式 1：通过 GitHub/GitLab 界面合并（推荐）
   # 方式 2：本地合并
   git checkout main
   git merge fix/[bug-id]-[brief-description]
   git push origin main
   ```

4. **清理**
   ```bash
   # 删除修复分支
   git branch -d fix/[bug-id]-[brief-description]
   git push origin --delete fix/[bug-id]-[brief-description]
   ```

5. **关闭 Issue**
   - 更新 Issue 状态
   - 添加修复说明
   - 关闭 Issue

**Anti-Patterns**：
- ✗ 审查未通过就合并
- ✗ 不处理审查意见
- ✗ 不删除已合并的分支
- ✗ 不关闭关联的 Issue

## 紧急修复（Hotfix）流程

对于紧急问题，可以简化流程：

1. **Diagnose**：快速定位问题（15 分钟内）
2. **Plan**：确定最小修复方案
3. **Implement**：实施修复
4. **Test**：运行关键测试
5. **PR**：创建紧急 PR
6. **Review**：快速审查（15 分钟内）
7. **Merge**：合并并发布

**关键原则**：
- 只修复问题，不做其他改动
- 测试必须通过
- 必须有审查（可以简化）
- 事后补充完整流程

## 状态管理

在修复过程中，跟踪当前状态：

```markdown
## 当前状态
- 阶段：Implement
- 进度：根因分析完成，正在实施修复
- 阻塞：无
- 预计完成：[时间]
```

## 阶段 Checklist 模板

每个阶段完成后，使用以下模板记录：

```markdown
## [阶段名称] 完成记录

### 完成时间
[YYYY-MM-DD HH:MM]

### 完成内容
- [ ] [检查项 1]
- [ ] [检查项 2]
- [ ] [检查项 3]

### 输出物
- [输出物 1]
- [输出物 2]

### 备注
[备注信息]
```

## Red Flags — STOP

当你有以下任何想法时，**停下来重新检查**：

- "一眼就能看出问题在哪"
- "直接改试试"
- "先 hotfix 再复盘"
- "这个问题以前修过"
- "加个 if 就行"
- "单元测试跑一下就行"

**所有这些都意味着你正在合理化跳过流程。回到 fix-bug 的第一步。**

## Common Rationalizations

| 借口 | 现实 |
|------|------|
| "问题太明显不用分析" | 明显的问题往往掩盖真正的根因 |
| "先 hotfix 之后再复盘" | 之后 = 永远不 |
| "我加个 if 就行" | if 补丁 = 技术债累积 |
| "这个问题以前修过" | 上次可能没修对，才复发 |
| "回归测试太麻烦" | 不写回归 = 同样的 bug 会再来 |
