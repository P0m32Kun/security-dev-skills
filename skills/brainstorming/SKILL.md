# 需求讨论与设计探索

> 在编码前通过协作对话将想法转化为完整设计。

## 前置条件

- 用户提出了一个初步想法或需求
- 需要进一步澄清和细化

## 核心原则

1. **一次一个问题** — 不要一次性问所有问题
2. **倾听 > 假设** — 不要假设用户想要什么，问清楚
3. **渐进式披露** — 从高层次开始，逐步深入细节
4. **记录决策** — 每个重要决策都要记录原因

## 流程

### 1. 了解项目背景

首先了解当前项目的上下文：
- 项目的技术栈
- 现有的架构和设计模式
- 相关的代码和文档

### 2. 澄清需求

通过提问来澄清需求：

- **目标**：你想要达成什么？
- **用户**：谁会使用这个功能？
- **场景**：在什么情况下使用？
- **约束**：有什么限制条件？
- **优先级**：哪些是必须的，哪些是可选的？

### 3. 提出方案

基于澄清的需求，提出 2-3 个可行方案：

```markdown
## 方案 A：[名称]
- 描述：...
- 优点：...
- 缺点：...
- 适用场景：...

## 方案 B：[名称]
- 描述：...
- 优点：...
- 缺点：...
- 适用场景：...

## 推荐
基于 [原因]，推荐方案 A/B。
```

### 4. 呈现设计

将最终设计以结构化的方式呈现：

```markdown
## 设计概览
- 目标：...
- 方案：...
- 影响范围：...

## 详细设计
### 组件/模块
- ...

### 接口/协议
- ...

### 数据流
- ...

### 错误处理
- ...
```

### 5. 编写设计文档

将设计保存到项目的适当位置：

```markdown
# [功能名称] 设计文档

## 背景
## 目标
## 方案
## 详细设计
## 影响范围
## 风险与缓解
## 时间线
```

### 6. 设计自检

- [ ] 需求是否完整理解？
- [ ] 方案是否覆盖所有场景？
- [ ] 是否考虑了边界情况？
- [ ] 是否评估了技术风险？
- [ ] 设计文档是否完整？

### 7. 过渡到实施计划

设计完成后，准备进入实施计划阶段：
- 确认设计已获批准
- 确认所有问题已解答
- 准备进入 writing-plans skill

## Anti-Patterns

- ✗ 不问就开始设计
- ✗ 一次性问太多问题
- ✗ 只给一个方案让用户没得选
- ✗ 设计太复杂，过度设计
- ✗ 不记录决策原因

## 完成标准

- [ ] 需求已完整澄清
- [ ] 至少 2 个方案已提出
- [ ] 最终方案已确定
- [ ] 设计文档已编写
- [ ] 用户已批准设计

## Red Flags — STOP

当你有以下任何想法时，**停下来重新检查**：

- "这个太简单，不用设计"
- "我已经知道怎么做了"
- "用户想要快速原型"
- "让我先写点代码再讨论"
- "想法已经很清楚了"
- "设计会拖慢进度"

**所有这些都意味着你正在合理化跳过流程。回到 brainstorming 的第一步。**

## Common Rationalizations

| 借口 | 现实 |
|------|------|
| "太简单不需要设计" | 简单项目正是未经审视的假设浪费最多的地方 |
| "先写点代码探索" | 探索可以，但要先呈现设计并获得批准 |
| "用户催我快点" | 用户的急 ≠ 跳过设计的理由 |
| "我已经做过类似的" | 每个项目都不同，验证假设 |
| "设计文档太重了" | 简单项目的设计可以只是几句话 |
