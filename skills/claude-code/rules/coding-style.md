# 编码风格规范

> 适用于所有 agent 生成和修改的代码。

## 核心原则

### 不可变性优先

- 优先使用 `const`、`final`、`immutable` 等不可变声明
- 避免原地修改数据结构，优先创建新实例
- 函数参数默认视为不可变，需要修改时显式复制

### KISS（Keep It Simple, Stupid）

- 解决当前问题，不做未来假设
- 优先选择读得懂的方案，而非"聪明"的方案
- 嵌套深度不超过 3 层，超过时提取函数
- 三元运算符只用于简单赋值，复杂逻辑用 if-else

### DRY（Don't Repeat Yourself）

- 重复代码出现第 3 次时提取
- 但不要为了 DRY 而引入不必要的抽象
- 如果两段代码"看起来相似"但"变化原因不同"，不要合并

### YAGNI（You Aren't Gonna Need It）

- 不实现"以后可能需要"的功能
- 不添加"以防万一"的参数、配置、扩展点
- 用到时再加，不要提前设计

## 文件组织

### 行数限制

| 范围 | 限制 | 动作 |
|------|------|------|
| 理想 | 200-400 行 | 正常 |
| 警告 | 400-600 行 | 考虑拆分 |
| 硬限制 | 800 行 | 必须拆分 |

### 文件结构顺序

1. 导入声明
2. 类型定义 / 接口
3. 常量
4. 主要逻辑（公共 API 在前）
5. 辅助函数
6. 内部实现细节

### 函数长度

- 理想：20-50 行
- 警告：50-80 行
- 硬限制：100 行 — 必须拆分

## 命名规范

### 通用规则

- 名字要能自解释，不要需要注释才能理解
- 缩写只用广泛认知的（`id`, `url`, `http`, `api`）
- 布尔值用 `is`/`has`/`should`/`can` 前缀
- 常量用 UPPER_SNAKE_CASE

### 风格一致性

- 同一文件/模块内命名风格必须一致
- 遵循语言惯例（Go: camelCase exported, Rust: snake_case, JS: camelCase）
- 不要混用风格（同一文件中不能有 `getUserName` 和 `fetch_user_data`）

## 错误处理

### 必须遵守

- 所有可能失败的操作必须有错误处理
- 错误信息包含操作上下文（做了什么、为什么失败）
- 不吞异常（空 catch 块）
- 不用 `try { ... } catch { return null }` 掩盖错误

### 错误信息格式

```
[操作] 失败: [原因]
  例如: "文件读取失败: 权限不足 (/etc/config.yml)"
```

### 错误传播

- 底层函数抛出具体错误
- 中间层决定是处理还是传播
- 顶层（调用方）负责用户友好的错误展示

## 质量底线

以下任何一条不达标，代码不能提交：

- [ ] 无编译/类型错误
- [ ] 无 linter 错误（警告需逐条评估）
- [ ] 新增代码有测试覆盖
- [ ] 公开 API 有类型标注
- [ ] 错误路径有处理
- [ ] 无硬编码的密钥、token、密码
- [ ] 无调试代码残留
