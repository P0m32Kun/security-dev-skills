# 模型路由决策

> 根据任务特征选择最合适的模型，在质量和成本之间取得最优平衡。

## 核心原则

1. **够用就好** — 不用 Opus 处理格式化任务
2. **复杂度匹配** — 模型能力与任务复杂度对齐
3. **成本意识** — 每次调用都有成本，选错模型的代价是 10-50 倍

## 模型特征速查

| 模型 | 定位 | 相对成本 | 适用场景 |
|------|------|---------|---------|
| Haiku | 快速、低成本 | 1x | 简单、模式化、高吞吐 |
| Sonnet | 平衡型 | 5x | 大多数开发任务 |
| Opus | 最强推理 | 25x | 复杂推理、架构设计、关键决策 |

## 决策矩阵

### 用 Haiku 的场景

- 格式化、linting、简单的文本转换
- 分类、标签、简单提取
- 代码模板生成
- 简单的文件操作（重命名、移动）
- 批量处理中的单个简单任务
- 简单的 Q&A、事实查询

**判断标准**：任务有明确的输入输出模式，不需要创造性推理。

### 用 Sonnet 的场景

- 常规功能开发
- Bug 修复（中等复杂度）
- 代码审查（常规）
- 编写测试
- 文档编写
- 重构（局部）
- 大多数 subagent 任务

**判断标准**：任务需要理解和推理，但有明确的上下文和约束。

### 用 Opus 的场景

- 系统架构设计
- 复杂 bug 的根因分析
- 跨多个文件/模块的重构
- 安全审计和漏洞分析
- 关键路径上的代码审查
- 需要深度推理的技术决策
- 复杂的多步骤计划制定
- 涉及权衡取舍的设计决策

**判断标准**：任务需要深度推理、多步规划、或需要创造性解决方案。

## Subagent 路由示例

```
主任务: "实现用户认证系统"

分解后的 subagent 路由:
├── [Opus] 设计认证架构（OAuth2 + JWT 方案选型）
├── [Sonnet] 实现登录接口
├── [Sonnet] 实现注册接口
├── [Sonnet] 实现 token 刷新逻辑
├── [Haiku] 生成 API 文档模板
├── [Sonnet] 编写集成测试
└── [Opus] 安全审查整体实现
```

## 决策流程图

```
任务到达
    │
    ├─ 是否需要创造性/深度推理？── 是 → Opus
    │
    ├─ 是否是模式化/简单任务？── 是 → Haiku
    │
    └─ 其他情况 → Sonnet
```

## 常见误判

| 误判 | 正确做法 |
|------|---------|
| 所有任务都用 Opus | 大多数任务 Sonnet 就够了 |
| 所有 subagent 用同一模型 | 根据子任务复杂度分别选择 |
| 简单任务用 Sonnet 是"保险" | 简单任务用 Sonnet 是浪费 |
| "反正便宜" 用 Haiku 处理复杂任务 | 质量不够返工更贵 |
| 安全相关任务用 Haiku 省钱 | 安全任务必须用 Opus |

## 成本参考

假设一次典型开发会话（20 次调用）：

- 全 Opus: ~500x 基准成本
- 全 Sonnet: ~100x 基准成本
- 优化路由 (3 Opus + 12 Sonnet + 5 Haiku): ~85x 基准成本

**优化路由比全 Sonnet 还便宜，同时关键任务用 Opus 保证质量。**
