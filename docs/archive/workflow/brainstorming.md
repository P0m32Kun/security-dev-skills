# 需求讨论与设计探索

> 通过协作对话，将模糊的想法转化为经过验证的设计方案。

## 前置条件
- 用户提出了一个需求或想法
- 项目已有基本的代码结构（可选）

<HARD-GATE>
**阻断条件**：设计未经用户批准，不能进入实施阶段。
无论需求看起来多简单，都必须经过设计讨论。
</HARD-GATE>

## 核心原则

1. **一次一个问题** — 不要用多个问题淹没用户
2. **优先选择题** — 能用选择题就不用开放题
3. **YAGNI** — 无情砍掉不必要的功能
4. **探索替代方案** — 至少提出 2-3 种方案再做决定
5. **增量验证** — 每个设计段落都获得用户认可后再继续

## 流程

### 1. 了解项目背景

在提问之前，先了解当前项目状态：
- 查看项目结构和关键文件
- 阅读最近的提交记录
- 了解现有的架构和约定
- 评估需求范围：如果涉及多个独立子系统，先帮助用户拆分

### 2. 澄清需求

一次一个问题，逐步明确：

**问题维度**：
- **目的**：解决什么问题？为谁解决？
- **约束**：技术限制、时间限制、环境限制
- **成功标准**：怎样算做完了？
- **边界**：什么不在范围内？

**提问技巧**：
- 优先使用选择题（A/B/C）
- 避免一次问太多
- 对模糊回答追问细节

**示例**：
```
✗ "你想要什么功能？" — 太开放

✓ "用户认证方案你倾向哪种？
   A) JWT Token（无状态，适合 API）
   B) Session + Cookie（有状态，适合传统 Web）
   C) OAuth2 第三方登录
   我推荐 A，因为你的项目是前后端分离架构。"
```

### 3. 提出方案

提出 2-3 种方案，每种包含：
- 核心思路
- 优势
- 劣势
- 适用场景

给出推荐意见和理由。

**示例**：
```markdown
## 方案对比

### 方案 A：JWT + Redis 黑名单
- 优势：无状态、易扩展
- 劣势：Token 一旦签发无法撤销
- 适用：API 服务、微服务架构

### 方案 B：Session + Redis
- 优势：服务端可控、易撤销
- 劣势：有状态、扩展需共享 Session
- 适用：传统 Web 应用

### 方案 C：OAuth2 + 第三方 IdP
- 优势：免去自建认证
- 劣势：依赖外部服务
- 适用：需要社交登录

**推荐方案 A**：你的项目是 API 服务，JWT 最合适。
```

### 4. 呈现设计

确认方案后，分段呈现设计细节：

**设计段落**（按复杂度调整篇幅）：
- 架构概览
- 模块划分与职责
- 数据流 / API 接口
- 错误处理策略
- 测试策略

**每段都需用户确认**：
> "以上是认证模块的架构设计，看起来对吗？有需要调整的地方吗？"

如果用户有异议，修改后重新确认。

### 5. 编写设计文档

设计获得批准后，输出设计文档：

```markdown
# [功能名称] 设计文档

> 日期：YYYY-MM-DD
> 状态：已批准

## 背景与目标
[需求背景和要解决的问题]

## 方案选择
[对比分析和最终选择]

## 详细设计
[架构、模块、接口、数据流]

## 测试策略
[测试层级和关键测试点]

## 变更记录
| 日期 | 版本 | 变更内容 |
|------|------|---------|
```

保存到 `docs/specs/YYYY-MM-DD-<topic>-design.md`（路径可按项目约定调整）。

### 6. 设计自检

写完设计文档后，自查：

- [ ] **占位符扫描**：是否有"待定"、"TODO"、未完成的部分？
- [ ] **内部一致性**：各段落之间是否矛盾？
- [ ] **范围检查**：是否聚焦到可以一次实施？
- [ ] **歧义检查**：是否有需求可以有两种理解？

发现问题直接修复。

### 7. 过渡到实施计划

设计文档通过后，进入实施计划阶段（`workflow/writing-plans.md`）。

## Anti-Patterns

- ✗ "这个太简单了，不需要设计" — 再简单也要过设计
- ✗ 一次问多个问题 — 用户会遗漏或敷衍
- ✗ 只提一个方案就决定 — 没有对比就没有判断
- ✗ 设计文档写完不给用户看 — 可能有误解
- ✗ 跳过设计直接编码 — 返工成本更高
- ✗ 设计过于详细，写了实施细节 — 设计关注"做什么"，不是"怎么做"

## 完成标准
- [ ] 项目背景已了解
- [ ] 需求已澄清（目的、约束、成功标准、边界）
- [ ] 至少 2-3 种方案已对比
- [ ] 设计分段呈现并获得用户批准
- [ ] 设计文档已输出并通过自检
- [ ] 用户已确认设计文档
