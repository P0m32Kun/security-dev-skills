# 标准化 Bug 修复流程

> 本文档是 Bug 修复的编排器，强制执行标准化流程，确保修复质量。

## 核心原则

1. **先诊断后治疗** — 必须先完成根因分析，再制定修复方案
2. **方案可选** — 提供 2-3 种修复方案，由用户决策
3. **计划先行** — 修复方案必须写成计划，按计划执行
4. **流程强制** — 每个阶段有明确的完成标准，不能跳过
5. **质量保证** — 必须有测试验证，必须有 Code Review

## 流程总览

```
Diagnose → Plan → Implement → Test → PR → Review → Merge
    │         │         │         │      │       │       │
  根因分析  修复计划   编码实施   测试验证  提交PR  代码审查  合并代码
```

## 阶段详情

### 阶段 1：Diagnose（诊断）

**目标**：完成根因分析，明确问题本质

**输入**：Bug 描述（现象、复现步骤、期望行为）

**输出**：根因分析报告

<HARD-GATE>
**阻断条件**：未完成根因分析不得进入 Plan 阶段

**检查项**：
- [ ] Bug 现象已明确描述
- [ ] 复现步骤已确认
- [ ] 问题代码位置已定位
- [ ] 根本原因已分析清楚
- [ ] 影响范围已评估
</HARD-GATE>

**流程**：
1. **理解 Bug 现象**
   - 记录 Bug 的具体表现
   - 确认复现步骤
   - 确认期望行为 vs 实际行为

2. **定位问题代码**
   - 使用日志、堆栈跟踪定位
   - 使用代码搜索工具（Semble、CodeGraph）辅助定位
   - 缩小问题范围到具体函数/模块

3. **分析根本原因**
   - 使用 5 Why 分析法追溯根因
   - 区分直接原因和根本原因
   - 分析是代码逻辑错误、边界条件、并发问题还是设计缺陷

4. **评估影响范围**
   - 哪些功能受影响
   - 哪些用户受影响
   - 是否有数据损坏风险
   - 是否需要紧急修复

**输出格式**：
```markdown
## 根因分析报告

### Bug 现象
- **表现**：[具体现象描述]
- **复现步骤**：[1. 2. 3.]
- **期望行为**：[应该是什么样]
- **实际行为**：[实际是什么样]

### 问题定位
- **文件**：[path/to/file.ext]
- **函数/方法**：[function_name]
- **代码位置**：[行号范围]

### 根本原因
- **直接原因**：[导致错误的直接代码]
- **根本原因**：[为什么会写出这样的代码/设计]
- **问题类型**：[逻辑错误/边界条件/并发问题/设计缺陷]

### 影响范围
- **受影响功能**：[功能列表]
- **受影响用户**：[用户范围]
- **数据风险**：[是否有数据损坏风险]
- **紧急程度**：[critical/high/medium/low]
```

**Anti-Patterns**：
- ✗ 不分析根因，直接开始改代码
- ✗ 只看表面现象，不深挖根本原因
- ✗ 只定位到文件，不精确定位到函数/行号
- ✗ 不评估影响范围，盲目修复

---

### 阶段 2：Plan（计划）

**目标**：制定修复方案，编写实施计划

**输入**：根因分析报告

**输出**：修复方案 + 实施计划

<HARD-GATE>
**阻断条件**：用户未确认修复方案不得进入 Implement 阶段

**检查项**：
- [ ] 已提供 2-3 种修复方案
- [ ] 每种方案的优缺点已说明
- [ ] 推荐方案已明确
- [ ] 用户已选择最终方案
- [ ] 实施计划已编写
</HARD-GATE>

**流程**：
1. **制定修复方案**（必须提供 2-3 种）

   对于每种方案，说明：
   - 修复思路
   - 优点
   - 缺点
   - 风险
   - 工作量估计

2. **明确推荐方案**
   - 基于修复效果、风险、工作量综合评估
   - 说明推荐理由

3. **用户决策**
   - 向用户展示所有方案
   - 等待用户选择最终方案
   - **不得自行决定使用哪种方案**

4. **编写实施计划**
   - 按照用户选择的方案，编写详细计划
   - 计划必须可执行、可验证

**输出格式**：
```markdown
## 修复方案

### 方案 A：[方案名称]
- **思路**：[修复思路]
- **优点**：[优点列表]
- **缺点**：[缺点列表]
- **风险**：[风险评估]
- **工作量**：[预估工作量]

### 方案 B：[方案名称]
- **思路**：[修复思路]
- **优点**：[优点列表]
- **缺点**：[缺点列表]
- **风险**：[风险评估]
- **工作量**：[预估工作量]

### 方案 C：[方案名称]（可选）
- **思路**：[修复思路]
- **优点**：[优点列表]
- **缺点**：[缺点列表]
- **风险**：[风险评估]
- **工作量**：[预估工作量]

### 推荐方案
**推荐使用方案 [X]**，原因：
- [推荐理由 1]
- [推荐理由 2]
- [推荐理由 3]

---

## 实施计划

基于用户选择的方案 [X]，制定以下计划：

### 步骤 1：[步骤名称]
- **目标**：[步骤目标]
- **操作**：[具体操作]
- **验证**：[如何验证完成]

### 步骤 2：[步骤名称]
- **目标**：[步骤目标]
- **操作**：[具体操作]
- **验证**：[如何验证完成]

### 步骤 N：...

### 完成标准
- [ ] [验收标准 1]
- [ ] [验收标准 2]
- [ ] [验收标准 3]
```

**Anti-Patterns**：
- ✗ 只提供一个方案，不给用户选择
- ✗ 不说明各方案的优缺点
- ✗ 不等用户确认就直接开始实施
- ✗ 实施计划过于笼统，无法执行

---

### 阶段 3：Implement（实施）

**目标**：按照计划实施修复

**输入**：实施计划

**输出**：修复代码

<HARD-GATE>
**阻断条件**：代码未通过编译不得进入 Test 阶段

**检查项**：
- [ ] 已创建修复分支
- [ ] 已按照计划逐步实施
- [ ] 代码编译通过
- [ ] 无语法错误
</HARD-GATE>

**流程**：
1. **创建修复分支**
   ```bash
   git checkout -b fix/[bug-id]-[brief-description]
   # 示例：fix/issue-123-null-pointer-in-user-service
   ```

2. **按照计划逐步实施**
   - 严格按照 Plan 阶段的计划执行
   - 每完成一步，标记进度
   - 如遇计划外情况，暂停并重新评估

3. **代码质量检查**
   - 代码编译通过
   - 无语法错误
   - 遵循项目代码规范

4. **自我 Review**
   - 检查是否真正修复了根因
   - 检查是否引入了新问题
   - 检查代码可读性和可维护性

**Anti-Patterns**：
- ✗ 不创建分支，直接在 main 上修改
- ✗ 偏离计划，随意修改
- ✗ 只改症状，不改根因
- ✗ 修改范围超出必要范围

---

### 阶段 4：Test（测试）

**目标**：验证修复有效，未引入新问题

**输入**：修复代码

**输出**：测试结果

<HARD-GATE>
**阻断条件**：测试不通过不得进入 PR 阶段

**检查项**：
- [ ] 原 Bug 已修复（可复现验证）
- [ ] 单元测试已通过
- [ ] 集成测试已通过
- [ ] 回归测试已通过
- [ ] 未引入新 Bug
</HARD-GATE>

**流程**：
1. **验证原 Bug 已修复**
   - 按照复现步骤验证
   - 确认期望行为已实现

2. **运行现有测试**
   ```bash
   # 运行单元测试
   npm test / pytest / go test / ...

   # 运行集成测试
   npm run test:integration / ...
   ```

3. **编写新测试**（如需要）
   - 为修复的代码添加单元测试
   - 覆盖边界条件
   - 防止问题再次出现

4. **回归测试**
   - 确认相关功能未受影响
   - 运行完整的测试套件

**输出格式**：
```markdown
## 测试报告

### Bug 修复验证
- [x] 原复现步骤已无法复现
- [x] 期望行为已实现

### 测试结果
- 单元测试：[X/Y 通过]
- 集成测试：[X/Y 通过]
- 回归测试：[X/Y 通过]

### 新增测试
- [是否新增测试]
- [测试覆盖的场景]
```

**Anti-Patterns**：
- ✗ 只手动测试一次，不运行自动化测试
- ✗ 不编写新测试防止回归
- ✗ 测试通过就认为完成，不做回归测试

---

### 阶段 5：PR（提交 Pull Request）

**目标**：提交 PR，等待 Review

**输入**：测试通过的修复代码

**输出**：PR 链接

<HARD-GATE>
**阻断条件**：PR 未创建不得进入 Review 阶段

**检查项**：
- [ ] 代码已提交到修复分支
- [ ] PR 已创建
- [ ] PR 描述完整（包含问题、原因、方案、测试）
</HARD-GATE>

**流程**：
1. **提交代码**
   ```bash
   git add .
   git commit -m "fix: [简要描述修复内容]"
   git push origin fix/[bug-id]-[brief-description]
   ```

2. **创建 PR**
   - 标题清晰描述修复内容
   - 描述包含完整的上下文

3. **PR 描述模板**
   ```markdown
   ## 问题描述
   [Bug 现象描述]

   ## 根本原因
   [根因分析]

   ## 修复方案
   [采用的修复方案及原因]

   ## 变更内容
   - [变更 1]
   - [变更 2]

   ## 测试
   - [测试方法]
   - [测试结果]

   ## 关联 Issue
   Fixes #[issue-number]
   ```

4. **请求 Review**
   - 指定 Reviewer（如项目有要求）
   - 说明需要关注的重点

**Anti-Patterns**：
- ✗ PR 描述过于简单，只有 "fix bug"
- ✗ 不关联相关 Issue
- ✗ 不说明测试情况

---

### 阶段 6：Review（代码审查）

**目标**：通过 Code Review，确保代码质量

**输入**：PR

**输出**：Review 通过

<HARD-GATE>
**阻断条件**：Review 未通过不得进入 Merge 阶段

**检查项**：
- [ ] 代码逻辑正确
- [ ] 代码风格符合规范
- [ ] 测试覆盖充分
- [ ] 无安全问题
- [ ] Reviewer 已批准
</HARD-GATE>

**流程**：
1. **自我 Review**
   - 在请求他人 Review 前，先自己 Review 一遍
   - 检查是否有遗漏

2. **响应 Review 评论**
   - 及时回复 Reviewer 的评论
   - 对合理建议进行修改
   - 对不同意见进行讨论

3. **修改代码**（如需要）
   - 根据 Review 反馈修改代码
   - 重新运行测试
   - 推送更新

4. **获得批准**
   - 至少一位 Reviewer 批准
   - 所有讨论已解决

**Anti-Patterns**：
- ✗ 不响应 Review 评论
- ✗ 对所有建议都无条件接受，不讨论
- ✗ Review 通过后不重新测试

---

### 阶段 7：Merge（合并）

**目标**：合并修复代码到主分支

**输入**：Review 通过的 PR

**输出**：代码已合并

<HARD-GATE>
**阻断条件**：合并前必须确认

**检查项**：
- [ ] CI/CD 已通过
- [ ] 无合并冲突
- [ ] 用户已确认合并
</HARD-GATE>

**流程**：
1. **确认 CI/CD 通过**
   - 检查所有自动化检查是否通过
   - 确认无构建错误

2. **解决冲突**（如有）
   - 拉取最新主分支
   - 解决合并冲突
   - 重新运行测试

3. **用户确认**
   - 向用户确认是否合并
   - **不得自行决定合并时机**

4. **执行合并**
   ```bash
   # 方式 1：通过 GitHub/GitLab 合并（推荐）
   # 方式 2：本地合并
   git checkout main
   git merge fix/[bug-id]-[brief-description]
   git push origin main
   ```

5. **清理**
   - 删除修复分支
   - 关闭相关 Issue

**Anti-Patterns**：
- ✗ 不等用户确认就合并
- ✗ 合并后不删除修复分支
- ✗ 不确认 CI/CD 状态就合并

## 紧急修复（Hotfix）流程

对于 **critical** 级别的 Bug，可以使用加急流程：

1. **跳过方案讨论**：直接采用最保守的修复方案
2. **简化计划**：计划可以更简洁
3. **加快 Review**：指定资深开发者快速 Review
4. **紧急合并**：用户确认后立即合并

但以下步骤**不能跳过**：
- 根因分析（即使是简化版）
- 测试验证
- Code Review
- 用户确认

---

## 状态管理

每个阶段有三种状态：
- **pending** — 未开始
- **in_progress** — 进行中
- **completed** — 已完成

状态转换规则：
- 前一阶段 completed 后，下一阶段才能 in_progress
- 阻断条件未满足时，不能标记为 completed
- 如发现问题，可回退到前一阶段
